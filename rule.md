# 让 AI 帮你学编程的提示原则

适用场景：从零开始学习任何编程技术（Maven、Spring、Docker、React……）

---

## 核心原则

**目标是看懂学会，不是看起来厉害。**

---

## 一、限制 AI 的行为

```
1. 不要炫技
   - 不用生僻 API、不用复杂的一行流、不展示"高级写法"
   - 用最朴素的写法，哪怕啰嗦一点

2. 不要过度抽象
   - 不要为了"设计模式"或"可扩展性"增加无关代码
   - 三行能解决的问题不要封装成一个类

3. 不要一次给太多
   - 每次只聚焦一个知识点
   - 复杂流程必须分步骤，每步只做一件事

4. 注释要言简意赅
   - 注释解释"为什么"，不复述"做了什么"
   - 不好的注释：// 返回用户列表
   - 好的注释：// scope=test 表示 junit 不会打包进最终 jar

5. 不要假设我已经知道
   - 第一次出现的概念必须解释
   - 缩写、术语要说明全称和含义
```

---

## 二、要求 AI 必须包含的内容

```
1. 最简可运行示例
   - 能跑起来的代码比完美的代码更重要
   - 先跑通，再理解，再扩展

2. 日常常用命令
   - 按使用频率排序，最常用的放最前面
   - 每条命令附一句话说明作用

3. 实际开发中常见问题
   - 不要只讲"正确用法"，要告诉我"哪里容易踩坑"
   - 出错时如何排查，按步骤写清楚

4. 对比说明
   - 有相似概念时，用对比表格或对比示例说清楚区别
   - 例：dependencies vs dependencyManagement，compile vs runtime
```

---

## 三、提问模板（可直接复制给 AI）

学习新技术时，用这个模板提问：

```
帮我学习 [技术名称]，我是初学者，请注意：
1. 不要炫技，代码用最简单直接的写法
2. 复杂流程分步骤说明，每步只做一件事
3. 注释解释"为什么"，不复述"做了什么"
4. 必须包含：日常常用命令 + 实际开发常见问题
5. 有相似概念时用对比说清楚区别
6. 先给我一个能跑起来的最小示例
```

---

## 四、学习路径原则

```
第一步：跑起来
  - 先让代码能运行，不追求完美
  - 不理解的部分先跳过，做个标记

第二步：看懂
  - 每行代码知道"大概在做什么"
  - 不需要理解所有细节，抓主干

第三步：改一改
  - 改一个参数，看结果有什么变化
  - 删掉一行，看报什么错
  - 动手改比反复阅读记得更牢

第四步：遇到问题再深挖
  - 报错时才去查报错原因，比预先学更有针对性
  - 把实际遇到的问题和解决方法记下来
```

---

## 五、判断 AI 给的内容是否合适

如果出现以下情况，让 AI 重新生成：

- 代码超过 50 行但你只问了一个小问题
- 出现你没见过的技术名词但没有解释
- 一次性涉及超过 3 个新概念
- 注释比代码还多，或注释只是复述代码
- 给了"最佳实践"但没给最简单的入门示例
